# ADR 0002: Разделение кабинетов и ролей

- Статус: accepted
- Дата: 2026-04-13

## Контекст

Система должна поддерживать три ключевые роли:

- системный администратор платформы;
- руководитель организации;
- сотрудник организации.

Требование бизнеса: у каждой роли должен быть отдельный кабинет с различным набором возможностей и своим UX.

## Решение

Принять следующую модель:

- `apps/web/admin-portal`: отдельное web-приложение для `platform_admin`;
- `apps/web/manager-cabinet`: отдельное web-приложение для `company_manager`;
- `apps/web/employee-cabinet`: отдельное web-приложение для `employee`;
- `apps/api/core-api`: единый backend с RBAC, tenant-aware политиками доступа и audit trail.

Ролевая модель:

- `platform_admin`: управляет всей платформой, организациями, глобальными настройками, аудитом, интеграциями и поддержкой.
- `company_manager`: управляет своей организацией, создает сотрудников, назначает доступы, видит данные только своего tenant.
- `employee`: работает только в рамках назначенных процессов и видит ограниченный набор данных.

## Последствия

Плюсы:

- четкая изоляция UX по ролям;
- проще управлять навигацией и доступом;
- легче проводить security review;
- можно независимо деплоить кабинеты.

Минусы:

- больше deployable units;
- потребуется shared UI-kit и общие auth flows;
- выше требования к управлению версиями frontend приложений.
