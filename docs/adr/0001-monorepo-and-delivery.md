# ADR 0001: Monorepo и delivery pipeline

- Статус: accepted
- Дата: 2026-04-13

## Контекст

Проект включает несколько independently deployable частей:

- публичный сайт;
- backend;
- web-кабинеты для разных ролей;
- мобильное приложение;
- чат-микросервис;
- инфраструктурные конфиги и документацию.

Нужно сократить стоимость поддержки, упростить совместное использование типов и стандартизировать delivery через GitHub.

## Решение

Использовать один монорепозиторий со следующими принципами:

- приложения лежат в `apps/`;
- отдельные микросервисы лежат в `services/`;
- общие библиотеки лежат в `packages/`;
- инфраструктура и CI/CD лежат в `infra/`;
- документация лежит в `docs/`.

CI/CD строится вокруг GitHub Actions:

1. PR pipeline проверяет lint, tests, typecheck и affected builds.
2. Merge pipeline собирает Docker images для deployable сервисов.
3. Образы публикуются в registry.
4. Публикация образа или успешный workflow вызывает deploy webhook.
5. Сервер получает сигнал и обновляет контейнеры через Docker.

## Последствия

Плюсы:

- единые стандарты;
- общие типы и SDK;
- проще поддерживать зависимости;
- одна точка входа для CI/CD;
- проще вести ADR/PDR рядом с кодом.

Минусы:

- требуется дисциплина по boundaries;
- CI нужно оптимизировать через affected builds и кэш;
- возрастает важность правил ownership и code review.
